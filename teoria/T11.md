## 1. Preparar para Producción

### Diferencias Development vs Production

| Aspecto | Development | Production |
|---------|-------------|------------|
| `NODE_ENV` | development | production |
| Logs | Verbose, queries | Solo errores |
| Errores | Stack traces | Mensajes genéricos |
| Hot reload | Sí | No |
| Minificación | No | Sí |
| HTTPS | Opcional | Obligatorio |
| Base de datos | Local/dev | Cloud/managed |

### Configurar NODE_ENV

```javascript
// src/app.js
const isProduction = process.env.NODE_ENV === 'production';

// Configurar según entorno
if (isProduction) {
  // Menos logs
  app.use(morgan('combined'));
} else {
  // Más verbose
  app.use(morgan('dev'));
}

// Error handler diferente
app.use((err, req, res, next) => {
  console.error(err);

  res.status(err.status || 500).json({
    error: true,
    message: isProduction ? 'Error interno' : err.message,
    ...(isProduction ? {} : { stack: err.stack })
  });
});
```

### Scripts de package.json

```json
{
  "scripts": {
    "dev": "node --watch --env-file=.env src/app.js",
    "start": "node src/app.js",
    "build": "echo 'No build step required'",
    "test": "node --test",
    "lint": "eslint ."
  },
  "engines": {
    "node": ">=22.0.0"
  }
}
```

### Estructura de archivos para deploy

```
proyecto/
├── src/
│   └── app.js
├── package.json
├── package-lock.json      # ✅ Commitear (versiones exactas)
├── .env                   # ❌ NO commitear
├── .env.example           # ✅ Commitear (plantilla)
├── .gitignore
├── Dockerfile             # Para Docker
├── docker-compose.yml     # Para desarrollo local
├── .dockerignore
└── .github/
    └── workflows/
        └── deploy.yml     # CI/CD
```

## 2. Variables de Entorno

### Buenas prácticas

```bash
# .env.example (commitear como referencia)
NODE_ENV=development
PORT=3000

# Base de datos
DATABASE_URL=mongodb://localhost:27017/myapp

# JWT
JWT_SECRET=cambia_esto_en_produccion
JWT_EXPIRES_IN=7d

# APIs externas
CLOUDINARY_URL=
STRIPE_SECRET_KEY=
```

### Validar variables con Zod

```javascript
// src/config/env.js
import { z } from 'zod';

const envSchema = z.object({
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
  PORT: z.coerce.number().default(3000),
  DATABASE_URL: z.string().url(),
  JWT_SECRET: z.string().min(32),
  JWT_EXPIRES_IN: z.string().default('7d'),
});

// Validar al iniciar - falla rápido si falta algo
const parsed = envSchema.safeParse(process.env);

if (!parsed.success) {
  console.error('❌ Variables de entorno inválidas:');
  console.error(parsed.error.flatten().fieldErrors);
  process.exit(1);
}

export const env = parsed.data;
```

### Usar en la aplicación

```javascript
// src/app.js
import { env } from './config/env.js';

app.listen(env.PORT, () => {
  console.log(`Servidor en puerto ${env.PORT} [${env.NODE_ENV}]`);
});
```

## 3. Docker

### ¿Por qué Docker?

- **Consistencia**: Mismo entorno en desarrollo, testing y producción
- **Aislamiento**: Dependencias no entran en conflicto
- **Portabilidad**: Funciona en cualquier máquina con Docker
- **Escalabilidad**: Fácil de replicar contenedores

### Instalar Docker

```bash
# macOS
brew install --cask docker

# Ubuntu
curl -fsSL https://get.docker.com | sh

# Windows
# Descargar Docker Desktop desde docker.com
```

### Dockerfile básico

```dockerfile
# Dockerfile

# Imagen base
FROM node:22-alpine

# Directorio de trabajo
WORKDIR /app

# Copiar archivos de dependencias
COPY package*.json ./

# Instalar dependencias (solo producción)
RUN npm ci --only=production

# Copiar código fuente
COPY . .

# Puerto que expone la app
EXPOSE 3000

# Usuario no-root por seguridad
USER node

# Comando para iniciar
CMD ["node", "src/app.js"]
```

### Dockerfile optimizado (multi-stage)

```dockerfile
# Dockerfile

# === Stage 1: Dependencias ===
FROM node:22-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

# === Stage 2: Producción ===
FROM node:22-alpine AS runner
WORKDIR /app

# Crear usuario no-root
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nodeapp

# Copiar dependencias del stage anterior
COPY --from=deps /app/node_modules ./node_modules
COPY --chown=nodeapp:nodejs . .

USER nodeapp

ENV NODE_ENV=production
EXPOSE 3000

CMD ["node", "src/app.js"]
```

### .dockerignore

```
node_modules
npm-debug.log
.env
.env.*
!.env.example
.git
.gitignore
*.md
Dockerfile
docker-compose*.yml
.dockerignore
coverage
.nyc_output
tests
```

### Comandos Docker básicos

```bash
# Construir imagen
docker build -t mi-api .

# Listar imágenes
docker images

# Ejecutar contenedor
docker run -d -p 3000:3000 --name mi-api mi-api

# Ver contenedores activos
docker ps

# Ver logs
docker logs mi-api
docker logs -f mi-api  # Follow

# Entrar al contenedor
docker exec -it mi-api sh

# Parar y eliminar
docker stop mi-api
docker rm mi-api

# Eliminar imagen
docker rmi mi-api
```

## 4. Docker Compose

### ¿Qué es Docker Compose?

Herramienta para definir y ejecutar aplicaciones multi-contenedor. Ideal para desarrollo local con base de datos, cache, etc.

### docker-compose.yml básico

```yaml
# docker-compose.yml
version: '3.8'

services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
      - DATABASE_URL=mongodb://mongo:27017/myapp
      - JWT_SECRET=dev-secret-32-characters-long!!
    depends_on:
      - mongo
    volumes:
      - .:/app
      - /app/node_modules
    command: npm run dev

  mongo:
    image: mongo:7
    ports:
      - "27017:27017"
    volumes:
      - mongo_data:/data/db

volumes:
  mongo_data:
```

### docker-compose para PostgreSQL

```yaml
version: '3.8'

services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgresql://postgres:postgres@db:5432/myapp
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:16-alpine
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=postgres
      - POSTGRES_DB=myapp
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  postgres_data:
```

### Comandos Docker Compose

```bash
# Iniciar servicios
docker-compose up

# Iniciar en background
docker-compose up -d

# Reconstruir imágenes
docker-compose up --build

# Ver logs
docker-compose logs -f

# Parar servicios
docker-compose down

# Parar y eliminar volúmenes
docker-compose down -v

# Ejecutar comando en servicio
docker-compose exec api npm test
```

## 5. CI/CD con GitHub Actions

### ¿Qué es CI/CD?

- **CI (Continuous Integration)**: Ejecutar tests automáticamente en cada push
- **CD (Continuous Deployment)**: Desplegar automáticamente cuando pasan los tests

### Workflow básico: Tests

```yaml
# .github/workflows/test.yml
name: Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout código
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - name: Instalar dependencias
        run: npm ci

      - name: Ejecutar linter
        run: npm run lint

      - name: Ejecutar tests
        run: npm test
        env:
          NODE_ENV: test
          JWT_SECRET: test-secret-32-characters-long!!
```

> 📂 **Código de ejemplo**: [`codigo/T11/.github/workflows/ci.yml`](../codigo/T11/.github/workflows/ci.yml) - Pipeline completo de CI/CD

### Workflow con MongoDB

```yaml
name: Tests con MongoDB

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      mongodb:
        image: mongo:7
        ports:
          - 27017:27017

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'

      - run: npm ci

      - run: npm test
        env:
          DATABASE_URL: mongodb://localhost:27017/test
          JWT_SECRET: test-secret-32-characters-long!!
```

### Workflow de Deploy

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm test

  deploy:
    needs: test
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      # Deploy a Railway
      - name: Deploy to Railway
        uses: bervProject/railway-deploy@main
        with:
          railway_token: ${{ secrets.RAILWAY_TOKEN }}
          service: mi-api
```

### Secrets en GitHub

1. Ve a tu repositorio → Settings → Secrets and variables → Actions
2. Añade secrets como `RAILWAY_TOKEN`, `RENDER_API_KEY`, etc.
3. Úsalos con `${{ secrets.NOMBRE_SECRET }}`

## 6. Deploy en Railway

### ¿Qué es Railway?

Plataforma PaaS moderna, similar a Heroku pero más económica. Free tier generoso.

### Pasos para deploy

1. **Crear cuenta** en [railway.app](https://railway.app)

2. **Conectar repositorio**
   - New Project → Deploy from GitHub repo
   - Seleccionar repositorio

3. **Configurar variables de entorno**
   - Variables → Add Variable
   - Añadir todas las variables de `.env.example`

4. **Añadir base de datos** (opcional)
   - New → Database → MongoDB/PostgreSQL
   - Railway genera `DATABASE_URL` automáticamente

5. **Configurar dominio**
   - Settings → Domains → Generate Domain

### railway.json (opcional)

```json
{
  "$schema": "https://railway.app/railway.schema.json",
  "build": {
    "builder": "NIXPACKS"
  },
  "deploy": {
    "startCommand": "npm start",
    "healthcheckPath": "/api/health",
    "restartPolicyType": "ON_FAILURE"
  }
}
```

### CLI de Railway

```bash
# Instalar
npm install -g @railway/cli

# Login
railway login

# Vincular proyecto
railway link

# Deploy manual
railway up

# Ver logs
railway logs

# Abrir en navegador
railway open

# Variables de entorno
railway variables
```

## 7. Deploy en Render

### ¿Qué es Render?

Alternativa a Heroku con free tier para web services y bases de datos.

### Pasos para deploy

1. **Crear cuenta** en [render.com](https://render.com)

2. **Nuevo Web Service**
   - New → Web Service
   - Conectar GitHub
   - Seleccionar repositorio

3. **Configurar**
   ```
   Name: mi-api
   Environment: Node
   Build Command: npm install
   Start Command: npm start
   ```

4. **Variables de entorno**
   - Environment → Add Environment Variable

5. **Base de datos**
   - New → PostgreSQL (free tier: 90 días)
   - Copiar `Internal Database URL`

### render.yaml (Infrastructure as Code)

```yaml
# render.yaml
services:
  - type: web
    name: mi-api
    env: node
    buildCommand: npm install
    startCommand: npm start
    envVars:
      - key: NODE_ENV
        value: production
      - key: DATABASE_URL
        fromDatabase:
          name: mi-db
          property: connectionString

databases:
  - name: mi-db
    databaseName: myapp
    user: myapp
```

## 8. Deploy en Fly.io

### ¿Qué es Fly.io?

Plataforma que ejecuta apps cerca de los usuarios (edge). Excelente para baja latencia.

### Instalación

```bash
# macOS
brew install flyctl

# Linux/WSL
curl -L https://fly.io/install.sh | sh

# Login
fly auth login
```

### Primer deploy

```bash
# Crear app (genera fly.toml y Dockerfile)
fly launch

# Deploy
fly deploy

# Ver logs
fly logs

# Abrir en navegador
fly open

# Ver estado
fly status
```

### fly.toml

```toml
# fly.toml
app = "mi-api"
primary_region = "mad"  # Madrid

[build]
  dockerfile = "Dockerfile"

[env]
  NODE_ENV = "production"
  PORT = "8080"

[http_service]
  internal_port = 8080
  force_https = true
  auto_stop_machines = true
  auto_start_machines = true

[[services]]
  protocol = "tcp"
  internal_port = 8080

  [[services.ports]]
    port = 80
    handlers = ["http"]

  [[services.ports]]
    port = 443
    handlers = ["tls", "http"]

  [[services.http_checks]]
    interval = "30s"
    timeout = "5s"
    path = "/api/health"
```

### Variables de entorno en Fly

```bash
# Establecer secreto
fly secrets set JWT_SECRET=mi-secreto-super-largo

# Ver secretos
fly secrets list

# Múltiples a la vez
fly secrets set DATABASE_URL=... JWT_SECRET=...
```

## 9. PM2: Gestión de Procesos

### ¿Qué es PM2?

Process Manager para Node.js en producción:
- Reinicio automático si crashea
- Cluster mode (múltiples instancias)
- Logs centralizados
- Monitorización

### Instalación

```bash
npm install -g pm2
```

### Comandos básicos

```bash
# Iniciar aplicación
pm2 start src/app.js --name "mi-api"

# Con variables de entorno
pm2 start src/app.js --name "mi-api" --env production

# Modo cluster (usa todos los CPUs)
pm2 start src/app.js -i max

# Listar procesos
pm2 list

# Ver logs
pm2 logs
pm2 logs mi-api

# Monitorizar
pm2 monit

# Reiniciar
pm2 restart mi-api

# Recargar sin downtime
pm2 reload mi-api

# Parar
pm2 stop mi-api

# Eliminar
pm2 delete mi-api

# Guardar configuración actual
pm2 save

# Restaurar al reiniciar servidor
pm2 startup
```

### ecosystem.config.js

```javascript
// ecosystem.config.js
module.exports = {
  apps: [{
    name: 'mi-api',
    script: 'src/app.js',
    instances: 'max',        // Cluster mode
    exec_mode: 'cluster',
    autorestart: true,
    watch: false,
    max_memory_restart: '1G',
    env: {
      NODE_ENV: 'development'
    },
    env_production: {
      NODE_ENV: 'production'
    }
  }]
};
```

```bash
# Usar con ecosystem
pm2 start ecosystem.config.js
pm2 start ecosystem.config.js --env production
```

### PM2 con Docker

Generalmente **no se usa PM2 dentro de Docker** porque:
- Docker ya reinicia contenedores
- Kubernetes/Docker Swarm escalan horizontalmente

En Docker, usa directamente:
```dockerfile
CMD ["node", "src/app.js"]
```

## 10. Logging y Monitorización

### Logging estructurado con Pino

```bash
npm install pino pino-pretty
```

```javascript
// src/utils/logger.js
import pino from 'pino';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  transport: process.env.NODE_ENV !== 'production'
    ? { target: 'pino-pretty' }
    : undefined
});

export default logger;
```

```javascript
// Uso en la aplicación
import logger from './utils/logger.js';

logger.info('Servidor iniciado');
logger.error({ err: error }, 'Error en petición');
logger.debug({ userId, action }, 'Usuario realizó acción');
```

### Middleware de logging

```javascript
// src/middleware/logger.middleware.js
import logger from '../utils/logger.js';

export const requestLogger = (req, res, next) => {
  const start = Date.now();

  res.on('finish', () => {
    const duration = Date.now() - start;
    logger.info({
      method: req.method,
      url: req.url,
      status: res.statusCode,
      duration: `${duration}ms`,
      userAgent: req.get('user-agent')
    });
  });

  next();
};
```

### Health Check endpoint

```javascript
// Endpoint para monitorización
app.get('/api/health', async (req, res) => {
  const healthcheck = {
    status: 'ok',
    timestamp: new Date().toISOString(),
    uptime: process.uptime(),
    environment: process.env.NODE_ENV
  };

  try {
    // Verificar conexión a BD
    await mongoose.connection.db.admin().ping();
    healthcheck.database = 'connected';
  } catch (error) {
    healthcheck.status = 'error';
    healthcheck.database = 'disconnected';
    return res.status(503).json(healthcheck);
  }

  res.json(healthcheck);
});
```

### Servicios de monitorización

| Servicio | Descripción | Free Tier |
|----------|-------------|-----------|
| **UptimeRobot** | Monitoriza uptime | 50 monitores |
| **Sentry** | Tracking de errores | 5K eventos/mes |
| **LogDNA/Papertrail** | Logs centralizados | Limitado |
| **New Relic** | APM completo | Limitado |

### Configurar Sentry

```bash
npm install @sentry/node
```

```javascript
// src/app.js
import * as Sentry from '@sentry/node';

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: 1.0
});

// Al inicio, antes de rutas
app.use(Sentry.Handlers.requestHandler());

// Al final, antes del error handler
app.use(Sentry.Handlers.errorHandler());
```

## 11. Checklist de Producción

### Antes de desplegar

```markdown
## Seguridad
- [ ] Variables sensibles en env vars (no hardcodeadas)
- [ ] HTTPS habilitado
- [ ] CORS configurado correctamente
- [ ] Helmet middleware activo
- [ ] Rate limiting configurado
- [ ] Validación de inputs en todos los endpoints

## Performance
- [ ] Índices de BD creados
- [ ] Queries optimizadas (no N+1)
- [ ] Compresión habilitada
- [ ] Archivos estáticos con caché

## Fiabilidad
- [ ] Health check endpoint
- [ ] Logs estructurados
- [ ] Manejo de errores robusto
- [ ] Graceful shutdown implementado

## DevOps
- [ ] NODE_ENV=production
- [ ] package-lock.json commiteado
- [ ] .env.example actualizado
- [ ] Tests pasando en CI
- [ ] Dockerfile probado localmente
```

### Graceful shutdown

```javascript
// src/app.js
const server = app.listen(PORT);

const shutdown = async (signal) => {
  console.log(`${signal} recibido. Cerrando servidor...`);

  server.close(async () => {
    console.log('Servidor HTTP cerrado');

    // Cerrar conexiones de BD
    await mongoose.connection.close();
    console.log('MongoDB desconectado');

    process.exit(0);
  });

  // Forzar cierre después de 10s
  setTimeout(() => {
    console.error('Forzando cierre');
    process.exit(1);
  }, 10000);
};

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```
